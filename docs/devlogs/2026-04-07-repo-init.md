# 2026-04-07 Repo Init

## 為什麼先做這個

新的 `estnet-globe-viewer` 要走 globe-first 正式交付線，所以第一個 commit 不能一邊建 repo、一邊偷塞 runtime、dataset、或 producer-specific 假設。

先做 repo init 有三個目的：

1. 把交付 repo 和 workspace-internal 文件邊界切乾淨。
2. 把第一個 commit 的責任收斂成「建立骨架」而不是「同時做半套 app」。
3. 為後續 `offline hero globe shell`、canonical truth、reference replay smoke 留下清楚的 commit 邊界。

## 這次建立了哪些檔案

### Repo Root

- `README.md`
  - 專案入口、目前狀態、第一個 commit 邊界與下一個 commit 目標
- `.gitignore`
  - 排除 build output、local env、暫存產物、workspace-internal 文件

### Formal Docs

- `docs/README.md`
  - 正式文件入口與分類規則
- `docs/architecture.md`
  - 專案角色、truth 邊界、依賴邊界
- `docs/sdd/service-driven-hero-globe-sdd-v1.md`
  - repo-local kickoff SDD
- `docs/phases/README.md`
  - phase 記錄入口
- `docs/phases/phase-00-repo-init.md`
  - 第一個 commit 的目標、輸出、排除項與驗證
- `docs/devlogs/README.md`
  - devlog 索引
- `docs/devlogs/2026-04-07-repo-init.md`
  - 本次工作說明

### Archive

- `archive/README.md`
  - 之後安置過時文件的規則

## 這次刻意沒有做什麼

這次沒有建立：

- `package.json`
- `src/`
- `public/`
- globe engine 或 renderer 選型
- mock truth provider
- replay adapter
- bootstrap-kit 依賴

原因很直接：這些都屬於下一個或再下一個 commit 的責任，現在放進來只會讓第一個 commit 邊界失焦。

## 目前沒有資料流

這個階段只有 repo 結構與文件，還沒有 runtime，因此也沒有 endpoint、satellite、timeline 或 provider 的資料流可描述。

## 驗證方式

本次驗證重點不是畫面，而是 repo hygiene：

1. repo 已建立並初始化為 `main`
2. `origin` 已設定
3. README 與 docs 入口可讀
4. authority、phase、devlog、archive 已分層
5. repo 內沒有 producer-specific 或 internal-only 內容
